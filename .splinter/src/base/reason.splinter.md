# src/base/reason.rs — commentary

- `remove_string_from_vec`: linear-scan defense in full — `by_from`/`by_to` values are an entity's adjacency list (its edge degree), kept small by design: clustering caps a kern at a bounded entity count and edges are pruned by decay/gc, so a list is tens of ids, not thousands. At that size a `Vec` scan beats a `HashSet` on cache locality and avoids per-edge hashing; and since the fields serialize (serde) into the bincode shards, `Vec<String>` → `HashSet<String>` is a persisted-format change to weigh against a nonexistent hot-path win.
- `move_entity`: the inline step narration (partition / stamp incoming / detach outgoing / restamp on reattach) duplicated the function doc's cross-kern policy verbatim and was removed; the `///` doc is the single authority.
- `remove_entity`: cascade step 1's old wording ("Fact entities are immune, preserving today's behaviour") predated the superseded-fact exception; the doc now matches the `is_fact() && !is_superseded()` check.

Second-pass migration:
- `superseded_ancestors`: this walk is how a bi-temporal `include_history` query reaches the Superseded revisions the ANN deliberately no longer holds (they are evicted at supersede time).
- `add_reason` (full idempotency rationale): `reasons` is a map, so re-insert is naturally idempotent; `by_from`/`by_to` are Vecs, so an idempotent re-observe (Rephrase edges, move_entity round-trips, re-ingest) would append a duplicate id, double-count it in `collect_reason_ids`, and leave a stale entry after `remove_reason` (which removes only the first occurrence). Test `add_reason_is_idempotent_on_adjacency` was card #56.
- `move_entity` cross-kern reason policy (moved from the doc): `Reason` tracks cross-kern targets via `to_kern_id` only — no `from_kern_id` mirror — so the invariant is: the kern hosting a reason also hosts its `from` endpoint. Outgoing reasons (`r.from == E`) travel with the entity; if their `to` is a third entity still in the source kern, stamp `to_kern_id = from_kern_id`; self-loops move intact, no stamp. Incoming reasons (`r.to == E`, `r.from != E`) stay in the source kern, stamped `to_kern_id = to_kern_id` (target moved). Both directions skip stamping when `to_net_id` is set (remote target must not be overwritten). Cascade order: (1) same-kern move = silent no-op; (2) entity removed from source `entities` (missing = silent no-op, matches `remove_entity`); (3) outgoing reasons detached from source maps, stamped, reattached to dst via `add_reason`; incoming stamped in place; (4) entity inserted into dst; (5) `entity_kern` repointed via `index_entity`; only relocated (outgoing) reasons get `reason_kern` repointed via `index_reason`. HNSW indices (`entity_idx`, `gnn_entity_idx`, `reason_idx`) and the lexical index key on entity/reason id only, so a move intentionally leaves them untouched.
- `remove_entity` cascade order (moved from the doc): (1) Active Facts immune, early return; Superseded facts lose immunity; (2) all incident reasons dropped from `kern.reasons`/`by_from`/`by_to` and from `g.reason_idx` + `reason_kern`; (3) entity dropped from `kern.entities`, `g.entity_idx`, `g.gnn_entity_idx`, `entity_kern`; (4) lexical index purged if installed. Single pass, no lock re-acquisition; infallible (silent no-ops), matching the previous `&mut Kern` signature semantics.
