# src/retrieval/cache.rs — commentary

Why the cache exists: retrieval wall-clock is dominated by two LLM calls — HyDE query expansion (~21 s on a local CPU model) and answer synthesis (~12 s) — while the graph traversal between them is sub-millisecond (see `kern profile`). A repeat/near-repeat query would pay ~30 s to reproduce an unchanged result. Cost of the global-epoch invalidation: any write flushes the cache; acceptable because for a memory daemon recall and ingest are distinct phases, so between writes the cache hits fully. Epoch bumps: every content mutation (entity placed/edited, reason added, kern registered/deregistered) routes through `get_mut`/`register`/`deregister`.

- `QueryCache::new`: before raising cap into the thousands, replace the VecDeque with a hash index on (tag, text_hash) plus a small vector ANN for the semantic path — the linear O(cap) scan is only negligible (sub-µs vs the ~30 s LLM path) at the default cap of 256.
Second-pass migration:

- `insert` epoch capture (backs the first half of the inline pointer): `epoch` is a parameter, not read from the graph inside `insert`, because it must be the epoch observed WHEN the result was computed. `answer::query_locked` captures it under the same read lock as the graph phase, so a write landing during the unlocked LLM phase leaves the entry born stale — it is inserted with the old epoch and every later lookup misses it. That is the correct outcome: the alternative (stamping "now" at insert time) would mint a fresh-looking entry holding pre-write results and serve them as current.
- `insert` empty-result skip (backs `result.entities.is_empty() -> return`): an empty result is never cached, so a query that found nothing is re-run rather than served from cache. Sound regardless of motive, since the epoch already guards correctness; the effect is that an empty entry cannot occupy an LRU slot or pin a "no results" answer for the lifetime of an epoch. NOTE: the original rationale for this line is not recorded anywhere in the source or the prior note — the above is reconstructed from the code. Treat as inferred, not as the author's stated intent.
- `lookup` / `lookup_text` both promote a hit to MRU and both require a matching `tag` and the current epoch; a stale-epoch entry is a miss and is left for LRU eviction rather than being actively purged, so `clear` on write is the only eager path.
