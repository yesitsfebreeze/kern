# splinter: src/retrieval/rerank.rs

Second-pass migration:

- `MAX_RERANK_CANDIDATES` (the inline `(see note)` pointer resolves here): a hard ceiling of 32 that clamps the user-configurable `rerank_pool_size` via the `.min()` chain in `llm_rerank`. Each candidate is rendered as `[i] <text truncated to 300 chars>`, so the cap bounds the candidate block at roughly 32 x 300 = ~9.6 KB (~2.4k tokens) before the instruction preamble and query. Without the clamp a large configured pool would put an unbounded number of candidates in the prompt and blow the context window. Pinned by `llm_rerank_caps_candidates_in_prompt_regardless_of_pool_size`, which asserts index 31 appears and index 32 never does at a configured pool of 1000.
- `parse_ranking` leniency contract: models emit both `1` and `1.0` for the same index, so whole-number floats are accepted (`whole_number_floats_accepted`). A *fractional* index (`1.5`) is not a formatting variant — it means the model did not produce an index list, so the whole ranking is discarded rather than partially parsed (`fractional_float_discards_ranking`). Out-of-range and negative indices are instead filtered individually: `-1 as usize` is huge and falls out of the `< pool` check, so it cannot panic on the `used[*i]` index.
- `llm_rerank` is fail-open at every step: disabled, no LLM, pool < 2, or an unparseable response all leave `results` in its incoming order. The reorder only permutes the first `pool` entries; the tail past the pool is split off and re-appended untouched, and any head index the model omitted is appended in original order, so no candidate is ever dropped by reranking.
- MAX_RERANK_CANDIDATES (32) is a hard prompt ceiling overriding cfg.rerank_pool_size: each candidate is ~300 chars, so a large configured pool would blow the LLM context window.
- parse_ranking accepts integer indices as 1 OR 1.0 (models emit both) but rejects fractions like 1.5. A fractional index invalidates the WHOLE ranking (returns None) — never trust a partial parse. Out-of-range indices (including a negative that wraps huge as usize) are filtered by the `< pool` check, no panic.
