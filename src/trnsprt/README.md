# `trnsprt`

kern's transport and typed-RPC layer. Two stacks, layered bottom-up:

```text
  services      search::SearchSvc        kern_rpc::KernRpc      (generated by service!)
  ───────────────────────────────────────────────────────────
  typed-RPC     typed::Channel<C> + Codec (JsonEnvelope | Bincode)   framed send/recv
                typed::Adapter (Inproc | stdio | TCP)
  ───────────────────────────────────────────────────────────
  MCP (legacy)  McpServer / Client / Registry / Transport      JSON-RPC, wire in `mod server`
```

## MCP stack (legacy)

The original boundary: client, server, and registry for the Model Context
Protocol. Two transports, one surface:

- **Stdio** (`ChildStdio`): JSON-RPC 2.0 newline-delimited UTF-8 frames over a
  child process's `stdin`/`stdout`. One complete JSON object per line;
  embedded newlines not permitted. **Not** LSP's `Content-Length` framing.
- **In-proc** (`InProcTransport` + an `InProcServer` impl): same JSON-RPC 2.0
  shape, no process, no pipe — for hot paths where stdio latency is a
  material cost.

Scope: spawn or wrap a server, run the `initialize` handshake +
`initialized` notification, `tools/list`, `tools/call`, reap the child on
shutdown. `Registry` keys live servers by `ServerId` and dispatches
`call_tool(id, name, args)` identically regardless of transport. The
JSON-RPC wire format is an implementation detail hidden inside `mod server`;
`serve_stdio` / `serve_rw` / `serve_http` run the server side.

## Typed-RPC stack

An additive, codec-framed `typed::Channel` over a `Transport` adapter
(`Inproc`, `stdio`, or `TCP`), with a pluggable `Codec` (`JsonEnvelope` or
`Bincode`). The `service!` macro (re-exported from
[`trnsprt-macros`](macros/)) generates a trait, a client, and a serve loop
per service from a single declaration. The two services:

- **`search`** (`SearchSvc`) — the search RPC surface.
- **`kern_rpc`** (`KernRpc`) — the kern daemon RPC surface, consumed by the
  CLI and other local clients over the per-cwd socket.

They share DTOs where their wire shapes overlap.

## Performance

In-proc per-call dispatch avoids the kernel pipe round-trip and UTF-8
re-parsing across a process boundary. The inline test
`inproc_transport_call_latency_under_ceiling` asserts a generous 5 ms
per-call ceiling (observed locally sub-100 µs); a child-process stdio
round-trip is typically two-to-three orders of magnitude slower due to
syscalls + scheduler wake-up.

## Extending

- New transports → implement `Transport` (read/write/kill) for the MCP stack,
  or a `typed::Adapter` for the typed stack.
- New services → one `service!` declaration generates trait + client + serve
  loop.
