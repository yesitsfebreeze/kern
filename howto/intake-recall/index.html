How-to

Intake & recall

Start a daemon, seed the graph, drop an intake delta, confirm it landed.

Before you start

kern on PATH, local Ollama with both default models:

Embedding dimension locks into the graph on first ingest; switching models later requires kern reembed. Not silent: store stamps its model, kern health prints MISMATCH beside embed:, wrong-dimension query vectors are dropped and counted.

1. Opt the project in

Pins to the nearest ancestor with .git or .kern. No config file required — create .kern/kern.toml only to override a default.

2. Start the daemon

Let your MCP client start it — kern mcp attaches to a running daemon, auto-spawns a detached one if none answers:

The daemon writes this entry into .mcp.json on startup if absent. To watch the first run (logs on stderr only; stdout is reserved for JSON-RPC):

3. Seed gravitons and claim kinds

Once per project, via MCP against the running daemon (the CLI can race a live daemon).

  • graviton tool, action: "add" per focus area — e.g. decisions, project state, preferences.
  • Seed text: example statements, one per line — each line embedded separately, mean-pooled into the pull vector; routes far better than an abstract description.
  • Optional mass (default 1.0): stronger attraction.
  • Unmatched claims → generic. See Acceptance & routing.
  • Built-in claim kinds: preference, decision, project, fact, code-fact, reference, procedural. Extras via claim_kind; registered kinds are offered to the distillation LLM alongside built-ins.

CLI equivalents, daemon stopped:

4. Drop a session into the intake

Anything writing a .txt delta into .kern/intake/ feeds kern — client hook, shell wrapper, your own script:

  • Daemon polls every 5s, distills each delta in one LLM pass, ingests the typed claims.
  • Moves to .kern/intake/done/ only once every claim committed; LLM outage → stays put, retried.
  • Archived deltas pruned after seven days.
  • Synchronous, already-typed ingest (skips distillation): ingest MCP tool or kern ingest.

5. Recall in the next session

query MCP tool: mode, kind, source, time ranges, min_conf, as_of (point-in-time), include_history (supersede chains). Returns scored passages, edges, chains — the agent synthesizes. See Retrieval.

Shell: kern search "indentation" --k 5 or kern query "what did we decide about indentation".

What a healthy graph looks like

  • reasons well above thoughts — claims connecting, not just arriving.
  • gravitons non-empty — (none) means seeding never happened; everything piles into generic.
  • unnamed small relative to thoughts — large share means gravitons don't cover the content.
  • done/ filling while .kern/intake/ stays near empty — drain keeping pace.
  • Non-zero panics/failures — something dying; the line beneath names the task.

When it does not work

Start with kern intake. A queued delta is retried forever rather than dropped, so a permanently stuck one looks exactly like a fresh one until you ask:

STUCK carries the last failure and is the whole diagnosis; wait means not picked up yet. kern intake drain forces one pass — routed to the daemon when one is serving so it never drains twice, in this process when none is — so you can drive the queue with no daemon at all and read the error immediately.

  • LLM endpoint unreachable. Transient: deltas queue and retry forever. kern intake shows them STUCK with the reason; confirm with curl http://localhost:11434/api/tags. WSL (NAT) with Ollama on the Windows host: localhost resolves inside the VM — pin the URL to the host gateway (ip route show default | awk '{print $3}') in .kern/kern.toml. Blanking both [reason].url and [embed].url disables distillation; the daemon logs a warning.
  • Nothing landing. Check in order: .kern/ exists where the daemon pinned; [intake].enabled is true; something is writing files. Non-UTF-8 → .kern/intake/failed/. Endpoint answers but drain never finishes: prose-not-JSON — kern intake names it.
  • Recall empty. kern health shows content: check MISMATCH beside embed: (fix: kern reembed). Otherwise widen the query — min_conf gates out fresh claims, which start at 0.6 until reinforced.
