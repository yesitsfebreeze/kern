# kern v2 — Native Specialist Training

> **Historical design sketch.** A forward-looking plan, not a description of
> shipped behavior — none of the phases below have been built. Kept as the
> rationale of record for the specialist-training direction.

**Goal:** extend kern from a self-maintaining memory substrate into a system that also *grows learned specialists* from its own graph — all inside the single Rust daemon. No Python, no subprocess, no dependency hell. You spawn it, it captures, it compacts, and over time it distills small LoRA specialists that ride alongside the memory.

The v1 daemon already owns the graph, the embeddings, the federation. v2 adds one new organ: a training loop that turns mature regions of the graph into adapter weights.

---

## The core idea

Three roles, cleanly separated:

- **kern (the graph)** — durable memory, self-compacting, per-cwd. Already built.
- **The teacher** — a strong inference LLM (a large hosted model, or a large local model) used *offline*, not at runtime. It reads mature regions of the graph and produces training examples: "given this kind of question about this domain, here's the grounded answer and the reasoning path through the graph."
- **The specialist** — a small base model + a LoRA adapter trained on the teacher's examples. It doesn't memorize the graph; it learns to *navigate* it — which gravitons matter, how reason-edges connect, when to defer to a `query` call. The graph stays the source of truth; the adapter learns the structure.

A specialist is born when an graviton (or a dense generic cluster) crosses a maturity threshold — enough thoughts, enough reason-edges, enough access heat. kern already detects this for clustering; v2 reuses that signal as the trigger to spin off a specialist.

---

## Phase 0 — Decide the Rust ML substrate

Before any code, pick the tensor/training library. This is the one irreversible choice.

1. **Evaluate `candle` vs `burn`.** candle (HF) is closer to the model ecosystem — GGUF loading, existing LoRA examples, qwen support. burn is a more complete training framework with a cleaner autograd story but a smaller model zoo. Lean candle unless burn's training ergonomics win decisively.
2. **Confirm three capabilities exist or are buildable:** (a) load your Ollama/qwen base weights, (b) attach trainable low-rank matrices to attention layers with the base frozen, (c) run a backward pass on CPU and on whatever GPU you have locally.
3. **Write one throwaway spike:** load a small qwen, attach a rank-8 LoRA to the q/v projections, overfit it on 20 hand-written examples, confirm loss drops and the adapter changes behavior. This de-risks everything downstream. Do not proceed to Phase 1 until this spike works.

---

## Phase 1 — Port LoRA training into kern (Rust)

This is the "translate the Python to Rust" step. A coding agent can do the bulk of it; you validate numerics.

1. **Reference the canonical implementation.** Point the agent at HF PEFT's LoRA (the `ΔW = BA` decomposition, frozen base, scaling by `α/r`) and the candle/burn LoRA examples. The math is small: two low-rank matrices per targeted layer, forward = `Wx + (α/r)·B(Ax)`.
2. **Implement the adapter module** in a new `src/train/` module: config (rank `r`, alpha, target layers, learning rate), the injected matrices, the modified forward pass.
3. **Implement the training loop:** batch examples → forward → cross-entropy loss → backward → optimizer step (AdamW is enough). Keep it deliberately minimal, matching kern's hand-rolled philosophy.
4. **Validate against a Python baseline.** Run the same tiny dataset + same seed through PyTorch PEFT and through your Rust loop. Loss curves should track closely. This is the test gate — numerical drift here means a silent bug.
5. **Output format:** save adapters as a small standalone file (safetensors-style) keyed by the graviton/cluster id they were trained for.

---

## Phase 2 — The teacher pipeline (graph → training data)

The specialist is only as good as the examples. This phase generates them from kern itself.

1. **Select a mature region.** When an graviton crosses the maturity threshold, snapshot its subgraph: thoughts, reason-edges, descriptors.
2. **Generate examples with the teacher.** For each region, prompt the strong LLM to produce (question, grounded-answer, reasoning-path) triples — where the reasoning path references actual node ids and reason-edges. Include negatives: questions where the right move is "query kern for X" rather than answer from memory. This is what teaches the specialist to defer instead of hallucinate.
3. **Hold out a validation slice** so you can measure each specialist later.
4. **Cache the dataset** under `.kern/train/<graviton-id>/` so retraining as the graph grows is incremental, not from scratch.

The teacher runs offline/async — never in the hot path. A `tick`-driven background job, same pattern as compaction.

---

## Phase 3 — Wire training into the daemon lifecycle

Make it invisible. The whole point is "you don't even realize it's working."

1. **New background job** alongside the existing tick: `maybe_train`. On each tick, check whether any region is (a) mature, (b) changed enough since its last adapter, (c) idle-compute available. If so, queue a training run.
2. **Train off-peak / rate-limited.** One region at a time, bounded CPU/GPU budget, fully cancellable — same fail-open discipline as the hooks. A killed training run just leaves the previous adapter in place.
3. **Version adapters** by content hash of their training set, so federation can dedupe and merge them the same way it merges thoughts. (A specialist becomes shareable: node A's adapter for "auth-domain" is fetchable by node B.)
4. **Register trained specialists** in the graph itself as a new node kind, so `health` can show what specialists exist and how fresh they are.

---

## Phase 4 — Inference: route through specialists

Now the lightweight-coordinator idea from our conversation becomes real.

1. **Coordinator stays dumb.** A small base model (or even rules + embeddings) maps an incoming query to the relevant graviton(s).
2. **Load the matching adapter(s)** onto the frozen base at query time — adapters are tiny, hot-swappable, and you can keep several resident.
3. **Specialist answers, grounded by `query`.** It uses its learned navigation to decide what to pull from kern, calls the existing MCP `query` tool, and synthesizes. The graph remains authoritative; the adapter just makes the model fluent in *this* project.
4. **Feedback loop:** wire bad answers back to the existing `degrade` tool and flag the region for retraining. The specialist improves as the graph and the usage signal improve — "everybody who uses it makes it better," locally first, then via federation.

---

## Phase 5 — Federation of specialists (stretch)

You already merge *facts* conflict-free via content-addressed CRDT. Extend the same idea to adapters.

1. **Share adapter weights** under their content-hash id over the existing gossip layer.
2. **Merge strategy:** adapters don't set-union like facts — averaging weights across divergent training sets is the catastrophic-forgetting trap. Safer first move: treat peer adapters as *candidates*, evaluate each against your local held-out validation slice, keep the best per region. Real weight-merging (task arithmetic, TIES-style) is a later research question, not a v2 requirement.
3. **Result:** a field's specialist improves across everyone running kern on that domain, with each node keeping whatever actually performs best for it.

---

## Sequencing & risk

- **Do Phase 0 first and honestly.** If the candle/burn spike is painful, the hybrid fallback (spawn a Python training subprocess, keep kern as orchestrator) is a legitimate v2 — you still get the single-entrypoint UX, just not the single binary. Don't let "pure Rust" block shipping.
- **Phases 1–4 are the real product.** Federation (Phase 5) is a bonus that rides on infrastructure you already proved.
- **Biggest risk is data quality, not Rust.** A mediocre teacher pipeline produces a confidently-wrong specialist. Invest in Phase 2's negatives and validation more than in training-loop elegance.
- **Keep the v1 invariant:** every new path fails open. A broken specialist or a failed training run must never degrade plain memory recall.

---

## What v2 proves

If it works, you've inverted the scaling paradigm in miniature: instead of one big model that knows everything, a swarm of cheap adapters over a shared self-maintaining graph, each fluent in one domain, each improving from use, each shareable. The teacher is rented; the memory is yours; the specialists are grown, not trained from scratch. That's the architecture you've been describing — kern v1 was the memory half, v2 is the learning half.
