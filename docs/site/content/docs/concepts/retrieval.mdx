title: "The retrieval pipeline"
---

This page follows one query from the text you type to the ranked thoughts that come back. Every stage is hand-written in `src/retrieval/` — there is no external ANN library, no reranking service, no vector database underneath — so each stage's cost and each stage's failure mode is something you can read.

The short version: kern seeds candidates from three independent sources (dense vectors, BM25, and a query-independent importance scan), fuses them by rank rather than by score, walks the reason graph outward from the fused seeds, then boosts, filters, and diversifies what the walk produced. Optional LLM steps (HyDE at the front, reranking and synthesis at the back) bracket the graph-only core.

## The shape of a query

The graph-only core is `retrieve_profiled` (`src/retrieval/answer.rs:172`). It is deliberately LLM-free, because the daemon holds a read lock on the graph for exactly that span and nothing else — every LLM call in the pipeline runs outside the lock (`src/retrieval/answer.rs:297`, `query_locked`).

## Stage by stage

### HyDE — repairing queries that are too short to embed well

A three-word query embeds badly: there is not enough text for the embedder to place it anywhere meaningful in vector space. `expand_query` (`src/retrieval/hyde.rs:4`) asks the LLM to write a paragraph that *would* answer the query, embeds that, and blends it with the real query vector at `hyde_fusion_weight` (default 0.5), re-normalizing the result.

The important detail, and one the README glosses: this fires **only for short queries**. The gate at `src/retrieval/hyde.rs:15` returns the query vector untouched once the query reaches `hyde_min_query_tokens` (default 6). A long query already carries enough signal; paying an LLM round-trip to paraphrase it would add latency and drift.

- **Adds:** usable vector geometry for queries that have none.
- **Costs:** one LLM generation plus one embedding call, on the queries that trigger it. It is the single most expensive stage when it runs.

### Dense seeding — approximate nearest neighbours over two vector spaces

`seed_with_important` (`src/retrieval/seed.rs:51`) reaches `search_all_unlocked` (`src/base/search.rs:57`), which runs the query against **two** indexes: the content-vector index and the GNN-vector index, then combines them in `merge_hits` (`src/base/search.rs:28`) at a fixed 0.4 content / 0.6 structure blend (`src/base/search.rs:25`).

When query options are active — a source, a kind, a confidence floor, a time window — the search switches to `search_all_filtered` (`src/base/search.rs:75`), which applies the predicate *during* the ANN traversal rather than after it. Post-filtering an unfiltered top-k is the classic filtered-ANN bug: ask for 15 and filter down to 2. The shared predicate lives in one place (`matches_keep`, `src/retrieval/seed.rs:77`) and is reused by the lexical seed and the post-filter so the three can never disagree.

- **Adds:** semantic recall — matches that share no words with the query.
- **Costs:** two graph traversals per query; the 2026-07-16 hot-path profiling pass attributed roughly 40 µs of a 103 µs query to this stage (the bench doc has since been retired along with the saturated trace — see the measurement note below).

<Callout type="info">
The 0.4/0.6 blend is a fixed constant, not a tuned one. It is an open question on the roadmap; treat it as a starting point rather than a measured optimum.
</Callout>

### Lexical seeding — the exact match vectors keep missing

BM25 over an in-RAM inverted index (`src/base/lexical.rs:62`, filtered variant at `:67`), wrapped by `seed_lexical` (`src/retrieval/seed.rs:86`). Dense retrieval is systematically bad at rare literal tokens — identifiers, error codes, proper nouns, version strings — because the embedder smooths them into their neighbourhood. BM25 is exactly good at those, and bad at everything dense retrieval is good at.

- **Adds:** exact-token recall that no embedding provides.
- **Costs:** an inverted index kept in RAM alongside the vectors, rebuilt from the graph.

### The importance scan — what the graph knows regardless of the query

`seed_important` (`src/retrieval/seed.rs:127`) does a parallel scan over every entity, keeping anything that clears a cosine floor (`important_min_cosine`, 0.20) and is either a `Fact` or has been accessed at least `important_access_threshold` (3) times. It is a third, query-*independent* candidate source: the things this graph has repeatedly proven useful.

Two honest caveats. It is O(N) on every retrieve — at corpus scale it needs an index, and this is the pipeline's clearest scaling gap. And because it is query-blind, it is down-weighted to `rrf_global_weight` (0.5) in fusion and deliberately excluded from PageRank's teleport set (`src/retrieval/answer.rs:122`), or a globally popular but irrelevant thought would ride to the top of every query.

- **Adds:** heat and durability as a retrieval signal, not just a ranking tiebreak.
- **Costs:** a full scan per query.

### PageRank — authority from graph structure

`pagerank` (`src/retrieval/pagerank.rs:28`) runs a **personalized** PageRank: the teleport vector is the query's own dense and lexical seeds, weighted by their scores (`teleport_vector`, `src/retrieval/pagerank.rs:5`), damping 0.85, up to 25 iterations with early exit once the rank vector stops moving. Dangling mass is redistributed along the teleport vector rather than uniformly (`src/retrieval/pagerank.rs:58`), which is what keeps the personalization from washing out.

This is what pure vector similarity structurally cannot do. Cosine distance asks "what looks like the query"; PageRank asks "what does the neighbourhood of things that look like the query keep pointing at". A well-cited hub two hops from every seed scores highly even if its own text matches nothing. The design rationale, including what was deliberately *not* built (DB-level authority, TrustRank seeding, Sybil defences), is in [PageRank for authority](../decisions/pagerank-authority.mdx).

### RRF fusion — combining lists whose scores mean different things

A BM25 score, a cosine similarity, and a PageRank mass are not comparable numbers. Reciprocal-rank fusion (`src/retrieval/fuse.rs:4`) sidesteps the problem by discarding the scores entirely and summing `weight / (k + rank)` across lists, with `rrf_k` = 60. Dense and lexical carry weight 1.0; the importance and PageRank lists carry `rrf_global_weight` = 0.5 (`src/retrieval/answer.rs:137`).

- **Adds:** a single ranking over four incommensurable sources, with no score normalization to tune or break.
- **Costs:** rank information is all it keeps — a dense hit that is *overwhelmingly* better than rank 2 gets no credit for the margin.

### Expansion — following the reasons

The seeds are a set of points. `expand` (`src/retrieval/expand.rs:178`) makes them a subgraph: a best-first beam walk out along reason edges from every seed, up to `max_expansions` (500), pruning any neighbour scoring below `global_best * decay` (0.25). Each surviving path is materialized as a `PathChain` — the actual sequence of entities and the labelled edges between them — which is what later gets rendered into the answer prompt as *why* these facts belong together.

Neighbour scoring (`score_neighbor`, `src/retrieval/expand.rs:305`) is a weighted sum of three terms: the neighbour's own content similarity to the query, the *edge's* vector similarity to the query, and the edge's own confidence plus a log traversal boost. Mode weights come from `Weights::for_mode` (`src/retrieval/seed.rs:37`).

- **Adds:** multi-hop reach and explanation. A fact connected to the query only through an intermediate is unreachable by any top-k over vectors.
- **Costs:** bounded work per query, but it is the stage that decides multi-hop quality — and per the LoCoMo baseline below, this is where kern currently measures worst.

### Merge, boosts, and gravity

`merge` (`src/retrieval/merge.rs:8`) pools seed and expansion scores per entity through a log-sum-exp accumulator, so an entity found by two independent routes earns roughly `+ln(2)`. The result is a magnitude, not a probability, and can exceed 1.0.

`apply_boosts` (`src/retrieval/score.rs:79`) then folds in the non-similarity signals: the score is multiplied by the entity's confidence, plus a QBST access/recency term (`qbst`, `src/retrieval/score.rs:61` — log access count and an exponential recency decay, capped), plus a flat bonus for `Fact` entities. `apply_gravity` (`src/retrieval/gravity.rs:9`) adds a pull toward the kern tree's gravitons, taking the **max** over gravitons rather than the sum so overlapping attractors never double-count. Gravity measured recall- and NDCG-neutral with about +7% p50 latency at five gravitons; it is a routing-coherence signal, not a quality win.

### Filtering, dedup, and MMR

`filter_delivery` (`src/retrieval/score.rs:119`) drops superseded entities (see [Time and supersession](./time.mdx)), applies the `min_deliver_score` floor — but only if at least one result clears it, so a hard query returns weak results rather than nothing — and truncates. With MMR enabled it truncates to the larger MMR pool, since truncating to the delivery cap first would leave MMR nothing to diversify.

`dedup_by_section` (`src/retrieval/diversify.rs:6`) collapses chunks of the same source section to their best representative. `mmr` (`src/retrieval/diversify.rs:46`) then does maximal marginal relevance over the pool: greedily pick the candidate maximizing `λ·sim(query) − (1−λ)·max sim(already picked)`.

<Callout type="warn">
`mmr_lambda` is the pipeline's sharpest quality lever. At the old default of 0.45, diversity outweighed relevance and MMR pushed legitimate near-duplicate cluster hits below rank 10 — recall@10 on the workload trace was 0.9250. Raising it to 0.75 took recall@10 to 1.0000 and NDCG@10 to 0.9987 on that same trace. Ingest-time dedup already removes true duplicates, so retrieval-time MMR does not need to sacrifice recall. The trace behind those numbers has since been retired (see the measurement note below); the `0.75` default stands.
</Callout>

### Rerank, synthesis, and cold backfill

`llm_rerank` (`src/retrieval/rerank.rs:11`) is off by default. When on, it hands the head of the list to the LLM and asks for a permutation of indices; a malformed response leaves the order untouched. The pool is hard-capped at 32 candidates (`src/retrieval/rerank.rs:9`) regardless of configuration, because ~300 characters per candidate would otherwise blow the context window.

`synthesize` (`src/retrieval/answer.rs:265`) builds the answer prompt from the path chains plus the top thoughts. Two abstention behaviours are worth knowing: the prompt instructs the model to reply with the exact `NO_ANSWER` string when the context does not contain the answer, and an empty context short-circuits to that string with no LLM call at all (`src/retrieval/answer.rs:282`) — cheaper and more reliable than asking a model to notice it has nothing.

Finally, if the hot graph returned fewer than `k` results, the MCP query tool backfills from the cold tier (`src/mcp/tools_query.rs:251`), a bounded brute-force cosine scan over LMDB (`src/base/store.rs:515`). Those hits are flagged `cold:true` so callers can tell.

### The query cache

`QueryCache` (`src/retrieval/cache.rs:20`) is an LRU of at most 256 entries keyed on the query vector, matched by cosine ≥ 0.97 — so a paraphrase hits the same entry. Correctness comes from a version stamp: every entry carries the graph's `mutation_epoch` at the moment it was computed, and any mismatch is a miss. Because the epoch is captured under the same read lock as retrieval, a write racing the unlocked LLM phase makes the new entry *born stale* — a miss on the next lookup, never a stale serve (`src/retrieval/cache.rs:75`).

## Index structures and their tradeoffs

Three indexes back the pipeline, behind a `VectorBackend` enum (`src/base/vector_backend.rs:8`) that lets a kern be resident or disk-spilled without the callers knowing.

**HNSW** (`src/base/hnsw.rs:25`) is the resident default: a hierarchical navigable small-world graph, deterministic-build and id-stable, supporting `insert`/`search` (`:140`, `:222`) and a genuinely pre-filtered `search_filtered` (`:247`). It trades memory for latency — every vector and every layer of the graph sits on the heap and is rebuilt from the graph on load, so RSS and load time grow with the kern without bound.

**DiskANN** (`src/base/diskann.rs:151` build, `:302` open, `:377` search) is the answer to that ceiling. It builds a disk-resident graph — degree 32, build list 64, alpha 1.2 (`src/base/diskann.rs:29`) — and mmaps it, taking the vectors off the resident heap. It engages only once a kern exceeds the configured `disk_threshold`, which defaults to never; small deployments are byte-for-byte unchanged. Post-snapshot writes buffer in an in-RAM delta with tombstones, folded back by a periodic consolidation task. The tradeoff is build time and a recall floor: the standalone index is held to recall@10 ≥ 0.90 against brute force. Rationale and remaining gaps — no product quantization yet, GNN and reason indexes still resident — are in [DiskANN-style spill](../decisions/diskann-spill.mdx).

**BM25** (`src/base/lexical.rs:24`) is an in-RAM inverted index with tunable `k1`/`b`, rebuilt from the graph. Cheap to build, cheap to query, and its memory is proportional to vocabulary rather than to dimensions.

## Quantization: what precision is actually given up

`QuantizedVec::encode` (`src/quant.rs:59`) supports three modes (`src/quant.rs:8`): `None` (f32, 4 bytes per dimension), `Int8` (1 byte), and `Binary` (0.125 bytes). Int8 uses a symmetric per-vector scale fixed at encode time, with distance computed directly on the codes (`quantized_cosine_distance`, `src/quant.rs:159`).

What you actually give up is measured, not assumed. `int8_recall_tracks_f64` (`src/base/hnsw.rs:874`) builds parallel f64 and int8 indexes over the same 300-vector corpus and asserts the top-10 sets agree on at least 75% of results across 25 queries. That is the guarantee: **int8 is not lossless, and the test pins the floor rather than claiming parity.** On the workload trace, int8 storage cut vector memory from 400 KiB to 100 KiB with the quality numbers unchanged.

Binary is implemented and tested (`src/base/hnsw.rs:909`) but deliberately **not user-selectable** — `QuantizationMode::parse` (`src/quant.rs:16`) rejects it, because its recall floor is too low without a rescoring pass that does not exist yet.

## What graph structure adds that vectors cannot

Two of the ranking signals come from graph shape rather than text.

The first is **PageRank authority**, above: a structural prior over what the graph collectively points at, personalized to this query's seeds.

The second is **learned re-embedding**. A background tick trains a small graph neural network per kern (`run_learned_propagation`, `src/gnn/propagate.rs:61`) — a 2-layer GCN, `dim → dim/2 → dim`, 24 epochs of Adam minimizing a link-prediction loss over real reason edges against sampled negatives. The output is blended with the input features at 0.6 self-weight, normalized, and written back as each entity's `gnn_vector`. It requires at least 128 thoughts in a kern to run at all.

That second vector is what the dense seed weights at 0.6. The effect is that two thoughts which never share vocabulary — so their content vectors are far apart — drift *toward each other in GNN space* if the graph keeps connecting them through reasons. Content embedding can only see the text of one thought at a time; this sees the shape around it. It is also the pipeline's most expensive background cost: training is synchronous on the tick, CPU-only, and per-kern rather than shared.

## What this actually measures

**No recorded retrieval baseline currently exists.** The synthetic workload
trace and the LoCoMo eval were both deleted on 2026-07-20: the LoCoMo score
collapsed ingest, retrieval and answering into one LLM-judged number dominated
by the answerer (a grounded run with the whole conversation in the prompt and
kern bypassed scored 0.187, versus 0.027 for kern on the same slice), and the
synthetic trace had saturated at recall@10 1.0 — no longer a hard enough test
to detect a regression. Numbers previously quoted here measured a pipeline
whose dominant term was not retrieval, so they are withdrawn rather than
restated. The replacement measures the retrieval layer alone (recall@k / MRR /
NDCG against per-turn evidence labels, no LLM in the loop) and is not built
yet; see the 2026-07-20 CHANGELOG entry.

<Callout type="warn">
The LoCoMo numbers come from a 7B judge at temperature 0, which is harsher than the lenient judges behind most published figures. That cuts both ways: the gap to 0.6 is real, but comparing a strict-judge number against a lenient-judge one is not a valid comparison in either direction. Never quote a retrieval number that did not come from a run.
</Callout>
