---
title: "Federation"
---

Federation lets several kern daemons on the same network share what they have
learned, with no coordinator and no primary replica. It is the least mature
subsystem in kern: parts of it work today, parts are wired but dead, and the
transport protects nothing.

<Callout type="warn" title="Status: `building`, not shipped">
Federation is marked `building` in the feature inventory. It is **off by
default** and should stay off unless you are deliberately experimenting.

**Works today:** peer heartbeats, multicast discovery, scope announcements,
entity-body sync into a quarantined remote namespace, CRDT counter /
last-write-wins deltas, and the single-entity fetch RPC that backs the
cross-network question path.

**Wired but not functional:** OR-Set deltas for `statements` — the sender
always emits an empty payload and the receiver rejects the target outright,
because content-addressed ids make imported statement text a forgery.

**Not built:** authentication, encryption, anti-entropy, and any bounded
guarantee that a cold entity eventually reaches every peer. The
[breakdown below](#what-actually-works-today) is exact.
</Callout>

## Why a daemon would want a peer at all

kern runs one daemon per working directory. That is deliberate — memory scoped
to a project stays relevant to that project, and the [graph](./graph.mdx) stays
small enough to reason over. But the boundary is wrong in one specific,
recurring case: **two machines doing the same work.**

The concrete motivating case is a laptop and a workstation, or two developers
on one service. You debug a deployment failure on the laptop at 11pm and kern
captures the cause — a config key that must be set before the migration runs.
The next morning you are on the workstation. That daemon has never seen the
incident, so you rediscover the same fact and it records a redundant copy of
knowledge the first daemon already holds.

Federation exists so the second daemon can answer that question. It is not
about scale or sharding a graph too large for one host. It is about a fact
learned once being available everywhere you work.

That framing sets the requirements. There is no central server to run, so the
design has to be leaderless. Daemons are laptops — they sleep, they miss
messages, they rejoin on a different network. So the merge has to tolerate
being offline, receiving messages out of order, and receiving the same message
twice.

## Discovery and gossip

A gossip-enabled node does three things on startup
(`src/commands.rs:937`): it binds a TCP listener for gossip messages, it starts
periodic announce loops, and — if discovery is on — it joins a UDP multicast
group.

**Discovery** broadcasts a plaintext string `kern:<network_id>:<tcp_addr>` to
`239.77.75.68` every ten seconds and listens for the same
(`src/gossip/discovery.rs:12` and `:40`). A peer is added only if its announced
`network_id` matches the local one (`src/gossip/discovery.rs:61`), which is how
two unrelated kern fleets on one office LAN avoid discovering each other. The
`network_id` defaults to a value generated per graph; you set a shared one in
config to make two daemons find each other
(`src/config/gossip.rs:11`).

Discovery is convenience, not the reliable path. Multicast is routinely dropped
by wireless APs, container bridges, and VPN interfaces. The path that is
verified to work is a manually seeded `peers` list in config — explicit
`host:port` entries the node dials directly. There is also an opt-in public
seed (`seed = true`, default `seed.kern.dev:7946`, `src/base/constants.rs:83`);
it stays off by default because dialing a public host from an unauthenticated
protocol must be a deliberate choice.

**Gossip** is push-based flooding with a fanout limit. Every message carries an
id; `broadcast` drops any message whose id it has already seen
(`src/gossip/node.rs:132`, backed by a TTL-bounded seen set at
`src/gossip/seen.rs:28`), then forwards to at most three randomly chosen peers
that are neither the sender nor itself (`src/gossip/node.rs:245`). Duplicate
suppression plus random fanout keeps a flood from becoming a storm; it is also
why the protocol is only *probabilistically* complete — nothing guarantees a
given message reaches a given peer.

## What crosses the wire

Frames are a big-endian `u32` length prefix followed by bincode
(`src/gossip/transport.rs:9`). Oversized prefixes are rejected before any buffer
is allocated (`src/gossip/transport.rs:23`), capped at 4 MB.

Seven message kinds exist (`src/gossip/types.rs:6`). Four carry real traffic:

- **PeerExchange** — the 30-second heartbeat. Each node sends its peer list;
  receivers adopt unknown addresses up to a 50-peer ceiling
  (`src/gossip/node.rs:82`).
- **Sphere** — the node's root scope: its graviton vector and text, and the
  radii that describe how wide its knowledge spreads
  (`src/gossip/handler.rs:107`). Receivers store it as a *phantom kern*, a
  read-only stand-in for a peer's region of knowledge.
- **EntitySync** — the payload that actually shares knowledge. Every heartbeat
  a node collects its non-remote entities, sorts by heat, and broadcasts the
  hottest 32 (`src/gossip/handler.rs:166`).
- **Delta** — incremental CRDT updates drained from the graph's pending queue
  (`src/gossip/handler.rs:191`), covering access counts, traversal counts,
  reason scores, and validity windows.

**Pulse** (activation spreading) and **Question** (an unresolved question
broadcast to peers, answered by whichever peer holds a close enough match) both
have live senders wired from the tick loop. **Fetch** backs the question path:
when a peer's answer references an entity body the local node lacks,
`spawn_fetch_entity` (`src/gossip/handler.rs:71`) issues a fetch, and the
holder answers it through the handler installed at startup
(`src/commands.rs:973`).

<Callout type="info" title="Top-32-by-heat is not replication">
A node only ever broadcasts its hottest entities. A fact that was captured
once and never recalled has low heat and may never be selected. There is no
anti-entropy pass, no merkle comparison, no catch-up snapshot — so
federation propagates what is *currently active*, not everything a peer
knows. Treat it as opportunistic sharing, not a backup.
</Callout>

## Why a CRDT and not a leader

Both obvious alternatives fail on the motivating case. A **leader** means one
designated daemon arbitrates writes — but the laptop is offline half the time,
so electing a leader among two sometimes-connected peers means either stalling
when the leader sleeps or building failover, a coordinator for two laptops. A
**lock** is worse: it puts a network round trip on the write path of a memory
system meant to be invisible, and when a peer is unreachable it has no correct
answer — block, or proceed and be inconsistent.

A CRDT sidesteps both by making the merge operation itself the guarantee. Every
federated field joins by an operation that is commutative, associative, and
idempotent (`src/base/merge.rs:87`):

| Field | Join |
|---|---|
| `access_count`, `traversal_count` | per-replica-slot max (G-Counter, `src/crdt.rs:29`) |
| `heat` | max — moot inbound, because remote heat is zeroed at the boundary (`src/base/merge.rs:20`) |
| `created_at` | min — the earliest observation wins |
| `accessed_at`, `updated_at` | max |
| `valid_until`, `Reason.score` | last-write-wins on `(lamport, producer)` |
| `statements` | **never imported** — content-addressing forbids it (`src/base/merge.rs:112`) |
| `status` | monotone toward `Superseded` — never back |
| `conf_alpha` / `conf_beta` | **never imported** |

Because every one of those is order-independent and duplicate-tolerant, the
gossip layer is allowed to be unreliable. Messages can arrive twice, arrive out
of order, or arrive weeks late after a laptop wakes up, and the result is the
same. That is the property you get: **strong eventual consistency** — any two
replicas that have received the same set of updates hold the same value,
regardless of the order they arrived in, with no coordination at any point.

Entity identity is what makes this hold at the graph level. Ids are content
hashes, so the same fact learned independently on two machines produces the same
id, and existence is plain set union — a peer can contribute an id but can never
mutate or delete one you already hold.

Confidence is the deliberate exception, and the reason is stated in the code at
`src/base/merge.rs:93`: a max-join on confidence would be an irreversible
poisoning pin. One peer asserting high confidence would permanently raise it
everywhere with no way back down. Trust stays replica-local — see
[Acceptance](./acceptance.mdx) for how a local daemon forms it.

## When two daemons learned contradicting facts

This is the interesting case, and the answer is that **the CRDT does not resolve
it — by design.**

Suppose A learned "the deploy script reads `DATABASE_URL`" and B learned "the
deploy script reads `DB_DSN`". Different content means different content hashes,
so these are two distinct entities, and set-union existence means both daemons
end up holding both claims. Nothing is overwritten or discarded.

That is correct, because a merge running on a background thread is the worst
possible place to adjudicate truth. It has no access to which claim is newer in
the real world, which source is more reliable, or whether both were true at
different times.

Contradiction is resolved by the same machinery that resolves it locally:
supersession and validity windows, described in
[Time & contradiction](./time.mdx). Federation only supplies two guarantees on
top of it:

- **Supersession is monotone and one-way.** Once a peer's copy is
  `Superseded`, the join marks the local copy `Superseded` and never flips it
  back (`src/base/merge.rs:95`). A superseded entity is also evicted from the
  vector indices on merge (`src/base/merge.rs:161`), so it stops surfacing in
  recall — the same invariant local supersession maintains.
- **`superseded_by` converges deterministically.** When two peers each nominate
  a different successor, the join takes the lexically greater id
  (`src/base/merge.rs:78`). This is a tiebreak chosen for determinism, not for
  semantics: it guarantees both replicas agree, not that they agree on the
  better answer.

And because confidence never crosses the wire, each daemon independently
decides which of the two contradicting claims it believes. Federation makes both
claims *visible* everywhere. It does not make belief uniform.

## Remote data is quarantined

Inbound entities never land in your own kerns. A node creates a phantom kern
named `remote-<sender network_id>-<sender kern_id>` and merges into that
(`src/gossip/handler.rs:476`). Every entity entering a remote kern first has
its ranking signals stripped — heat and access counts zeroed, confidence reset
to the neutral prior (`src/base/merge.rs:20`) — so a peer's assertions never
buy rank locally. Three rules bound the merge (`src/base/merge.rs:136`):
an id already owned by a *different* kern is rejected outright as a hijack
attempt; a genuinely new id is inserted only while the remote kern is under its
50,000-entity cap; and an id already present in the target is CRDT-joined. The
cap has no backpressure — at the ceiling, new remote ids are dropped while known
ones keep merging.

<Callout type="error" title="The transport is unauthenticated and unencrypted">
There is no encryption. Gossip is raw TCP; discovery is plaintext UDP.
There is no peer authentication and no payload signature — a node cannot
tell which peer authored anything it receives.

**In practice this means: anyone who can reach the gossip port can read
your entire federated graph in cleartext, and can inject arbitrary entities
into it.** Anyone who can sniff the segment sees every shared thought pass
by. The `network_id` is not a credential — it is broadcast in the clear in
every discovery packet, so anyone on the segment can read it and join that
network.

**Acceptable placement:** a network segment where you trust every host — a
home or lab LAN, a private VPC subnet, or a WireGuard mesh. Firewall both
the gossip TCP port and the UDP discovery port to that segment, and on a
multi-homed host bind `addr` to a specific private interface rather than
the default `0.0.0.0:7400`.

**Unacceptable placement:** a public interface, a coffee-shop or hotel
network, a shared office VLAN, or anything reachable from the internet.

If you need confidentiality or peer authentication today, provide it at the
network layer by running gossip only inside a VPN mesh. The protocol
provides neither, and there is no configuration that makes it do so.

The full trust model, including exactly what a malicious peer can and
cannot do, is in [Security](./security.mdx).
</Callout>

## What actually works today

| Capability | State |
|---|---|
| Off by default, no port bound | works (`src/config/gossip.rs:62`) |
| Manually seeded `peers` | works — the reliable path |
| Multicast discovery, same `network_id` | works when the network passes multicast |
| Peer heartbeat + peer exchange | works |
| Scope (Sphere) announce | works |
| Entity-body sync, hottest 32 per heartbeat | works |
| CRDT counter + LWW deltas | works |
| OR-Set deltas for `statements` | **dead by design** — sender emits empty (`src/gossip/handler.rs:216`), receiver rejects the target (`src/gossip/handler.rs:448`) |
| Single-entity fetch RPC | works — handler wired at startup (`src/commands.rs:973`), issued by the question path |
| Anti-entropy / catch-up for cold entities | **not built** |
| Authentication, encryption, signatures | **not built** |

One more caveat on the CRDT layer: `src/crdt.rs` ships only `GCounter` and the
`lww_wins` comparator. The last-write-wins joins described above are applied by
hand in `src/base/merge.rs` and `src/gossip/handler.rs:359` rather than
expressed as reusable CRDT types, and no OR-Set type exists at all. The
semantics hold, but they are enforced by hand at each call site, not by a type.

## Turning it on

See [Configure kern](../howto/configure.mdx) for the `[gossip]` block: set
`enabled = true`, set a shared `network_id` across the daemons you want paired,
and prefer an explicit `peers` list over relying on discovery. Then re-read the
danger admonition above and confirm the segment qualifies. For where federation
sits relative to the intake, the tick, and recall, see
[Architecture](./architecture.mdx).
