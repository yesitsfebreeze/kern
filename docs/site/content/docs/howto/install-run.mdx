---
title: "Install & run the daemon"
---

By the end of this page you will have the `kern` binary on your PATH, a daemon
running for one project directory, and a `.kern/` directory on disk holding that
project's graph. You will also know how to tell whether it is healthy, how to
stop it, and how to start over.

## Prerequisites

kern calls a local [Ollama](https://ollama.com) for embeddings, distillation and
answers. Pull the default models before the first run:

```bash
ollama pull qwen3-embedding:0.6b   # embeddings
ollama pull granite4:3b            # distillation / reasoning, and answers
```

The answer model aliases the reason default and falls back to the reason
endpoint, so two pulls cover all three legs.

You can point kern at any OpenAI-compatible endpoint instead — see
[Configure models](./configure.mdx).

## 1. Install the binary

```bash
# Linux / macOS
curl -fsSL https://raw.githubusercontent.com/yesitsfebreeze/kern/master/install.sh | sh
```

```powershell
# Windows (PowerShell)
irm https://raw.githubusercontent.com/yesitsfebreeze/kern/master/install.ps1 | iex
```

Both installers fetch the prebuilt release binary for your platform and print
the install path. To build from source instead, with a Rust toolchain:

```bash
cargo build --release   # -> target/release/kern
# or
cargo install --path .
```

## 2. Opt the project in

kern gates on a `.kern/` directory. Create it in the project you want to have a
memory:

```bash
cd /path/to/your/project
mkdir .kern
```

The daemon also creates `.kern/` on its first persist, but making it yourself
means intake and recall are active from the very first session rather than
after the first flush.

<Callout type="info">
The binary walks up from your cwd to the nearest ancestor holding `.git`
(falling back to the nearest holding `.kern/`) and re-pins itself there before
loading anything — `Config::resolve_root`, `src/config/mod.rs`. Launching from a
subdirectory therefore attaches to the project's graph instead of booting an
empty one.
</Callout>

## 3. Start the daemon

```bash
kern --daemon      # or: kern daemon
```

It prints `kern running in daemon mode (ctrl-c to stop)` and stays in the
foreground. To run it detached and keep the logs:

```bash
kern --daemon > .kern/daemon.log 2>&1 &
```

To serve MCP over HTTP as well as the local socket, add an address:

```bash
kern --daemon --mcp-addr 127.0.0.1:3000
```

You rarely need to do this by hand. `kern mcp` attaches to a running daemon and
auto-spawns a detached one if there is none (`src/commands/mcp_cmd.rs:11`), so
registering the MCP server is usually the only step — see
[Wire up MCP](./mcp.mdx).

<Callout type="warn" title="A bad config stops the boot">
Every `kern` invocation loads the config before it does anything else. An
unparseable `kern.toml`, or one that fails validation, prints the offending key
to stderr and exits **78** (`EX_CONFIG`) — it does not warn and continue
(`src/main.rs:16`). An *absent* config is legitimate and boots on defaults, so
deleting the file is a valid fix. `--help` and `--version` still answer, because
argument parsing runs first.
</Callout>

## Where state lives

Everything is under `<project>/.kern/`:

| Path | What it is |
| --- | --- |
| `.kern/data/data.mdb` | The LMDB environment: hot graph plus cold tier. |
| `.kern/data/lock.mdb` | LMDB's lock file. Never edit or delete it while running. |
| `.kern/intake/*` | Dropped files waiting to be drained: `.txt` distilled as transcripts, anything else stored as documents. |
| `.kern/kern.toml` | Optional config overrides. Absent by default. |
| `.kern/data/mcp-token` | Bearer token for the HTTP/SSE surface, minted on first use. |
| `.kern/data/logs/*.log` | stdout+stderr of detached children: `daemon.log`, `hub.log`. Owner-only, append-only. |

The intake path is cwd-relative by contract and is *not* derived from
`data_dir` (`src/config/intake.rs`) — the MCP server resolves it against the
session's working directory.

The local RPC endpoint lives outside the project: `$XDG_RUNTIME_DIR/kern-<tag>.sock`
(or `/tmp/kern-<user>-<tag>.sock`) on Unix, and the named pipe
`\\.\pipe\kern-<user>-<tag>` on Windows, where `<tag>` is a hash of the canonical
cwd — `Endpoint::kern`, `src/trnsprt/src/typed/local.rs:16`.

## One daemon per working directory

The socket name is derived from the cwd hash, so each project gets its own
daemon and its own graph, and several can run on one host. A second `kern --daemon`
in the *same* directory loses the bind race and exits immediately:

```
kern: another daemon already running at /run/user/1000/kern-3f2a91c4.sock — exiting
```

That check happens before any further scaffolding spins up
(`src/commands.rs:736-745`), so a duplicate start is cheap and harmless.

## Is it healthy?

Ask the running daemon through MCP, or run `kern health`. Both report the same
`HealthStats` (`src/mcp.rs:117`, `src/commands/admin.rs:33`): graviton, kern,
entity, reason, unnamed and claim-kind counts, plus the degradation signals —
tick queue depth, tasks done and average latency, `task_panics` and
`task_failures` with the most recent of each, `cold_evicted`, and the stored
`embed_model` / `embed_dim` with a mismatch flag.

Read it as follows. Non-zero entities means the graph has content. A growing
`unnamed` count means clustering has work queued. A non-zero panic or failure
count means maintenance is degraded — it kept running, which is the point, but
something is dying every beat. A climbing `cold_evicted` means the 50k cold cap
is dropping the oldest non-durable thoughts for good. `MISMATCH` on the `embed:`
line means the index was built with a different model than the one configured;
run `kern reembed`.

`kern health` reads the on-disk graph directly and asks the daemon only for the
tick figures. With no daemon serving this directory it prints
`tick: (no daemon serving this directory)` rather than blocking.

Two more signals that need no client:

- `.kern/intake/` is draining rather than accumulating. Files that pile up mean
  distillation is failing — usually a missing `[reason]` model. Nothing is lost:
  a delta stays queued until a pass succeeds.

Logs go to **stderr** via `tracing`, filtered by `RUST_LOG` and defaulting to
`warn` (`src/main.rs:27`). Turn them up when something looks stuck:

```bash
RUST_LOG=kern=info kern --daemon
```

The targets worth knowing: `kern.kern_rpc` (socket bind), `kern.intake`
(intake), `kern.persist` (a refused stale flush), `kern.mcp` (proxy vs
standalone), `kern.tick` (a maintenance task that panicked), `kern.store` (cold
evictions, an embedding-model mismatch).

A **detached** child keeps its output. `kern mcp` spawning a daemon, and the hub
spawning a per-root node, both redirect stdout and stderr into
`<data_dir>/logs/<arg>.log` — `.kern/data/logs/daemon.log` and
`.kern/data/logs/hub.log` by default (`src/config/detached_log.rs:23`,
`Config::log_dir`, `src/config/mod.rs`). The files are opened owner-only
(`0600`) and **appended**, never truncated, so a restart does not erase the log
explaining why it restarted. If the log cannot be opened the spawn still
happens — the child's output is discarded and the reason is printed on the
parent's stderr.

<Callout type="warn" title="There is no `kern status`">
Nothing tells you whether a daemon is running for this directory. Check for
the socket (`ls $XDG_RUNTIME_DIR/kern-*.sock`) or the process
(`pgrep -af 'kern --daemon'`) instead.

Worse, the `kern <subcommand>` CLI does not talk to the daemon — it opens the
on-disk graph directly and **can race a live daemon**. `kern health`,
`kern query` and friends may read a stale snapshot, and writing subcommands
can contend with the daemon's writer. A guarded-flush protocol keeps a stale
snapshot from overwriting newer on-disk state, but the safe rule stands:
**use the MCP tools for live state, and stop the daemon before offline
commands** (`kern reembed`, `kern compact`, `kern migrate`).
</Callout>

## Stopping it

Ctrl-C in the foreground, or `pkill -f 'kern --daemon'` when detached. On
shutdown the daemon runs one final guarded flush and prints `shutting down...`
then `done` (`src/commands.rs:788`). Let it finish — killing with `SIGKILL`
during the flush drops whatever was still only in RAM.

## Starting over

With the daemon stopped:

```bash
rm -rf .kern/data          # wipe the graph, keep config and queued intake files
rm -rf .kern               # wipe everything, including opt-in
```

Deleting rows frees pages inside `data.mdb` but LMDB only returns that space to
the OS on a compaction. The file shrinks on the next daemon restart, which
auto-compacts, or offline via `kern compact`.

Upgrading from a pre-LMDB build that kept one bincode shard per kern? Run the
importer once per data directory, daemon stopped:

```bash
kern migrate               # migrates <cwd>/.kern/data/ in place
kern migrate /dir          # or a specific data directory
```

New projects need no migration.
