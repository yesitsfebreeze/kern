---
title: "DiskANN spill for large kerns"
---

**When a kern's entity count crosses a configured threshold, its vector index
spills from resident HNSW to a memory-mapped DiskANN (Vamana) snapshot with an
in-RAM write delta — instead of sharding, quantizing harder, or evicting.**
Status: shipped and wired into live search, but off by default —
`disk_threshold` defaults to "never spill" (`src/config/graph.rs:20`), so small
deployments run byte-for-byte the resident path. The spill covers the entity
content index only.

## The constraint

The resident index keeps every entity vector and every layer of the HNSW graph
on the heap, rebuilt from the graph on load — RSS and load time grow with the
kern without bound, and no entity-eviction cap ships to stop it. That is
deliberate at small scale (see the index comparison in
[The retrieval pipeline](../concepts/retrieval.mdx)), but it makes corpus size
a hard wall: a kern with millions of entities eventually will not load on the
machine it lives on. Something has to take vectors off the resident heap
without giving up approximate-nearest-neighbour latency, and it has to do so
under kern's ground rules — no external ANN library, no service dependency,
continuous ingest that never pauses for a rebuild.

## Alternatives considered

**An entity-eviction cap.** Bound the resident set by dropping cold entities.
Rejected in the design itself: `src/config/graph.rs` documents that huge-corpus
scaling is "the DiskANN index's job, not this cap". Eviction silently deletes
recall — a fact captured once and never yet recalled is exactly the fact a
memory system exists to keep — and choosing victims well is the same hard
problem as retrieval.

**Harder quantization in RAM.** Int8 already ships and cuts vector memory 4x;
binary exists but is deliberately not user-selectable because its recall floor
is too low without a rescoring pass that does not exist
(`src/quant.rs:20`). Quantization shrinks the constant; it does not remove the
`O(N)` heap term, and the HNSW graph structure itself still grows with N.

**An external vector database or ANN library.** Rejected on the project's
standing constraint: every retrieval stage is hand-written and readable
(`src/retrieval/`, `src/base/hnsw.rs`), so each stage's cost and failure mode
is inspectable. A service dependency also contradicts the one-daemon-per-
directory deployment model.

**The cold tier as the answer.** The original plan's first phase targeted the
cold tier's O(n) JSONL scan. That problem dissolved before the wiring landed:
the cold tier moved into LMDB with int8-on-disk vectors and a bounded scan
capped at 50,000 entries (`src/base/store.rs:515`,
`src/base/constants.rs:17`), so cold recall no longer degrades linearly. What
remained — and what DiskANN actually fixes — is the **hot, resident** ceiling.

## Why Vamana-on-disk wins

DiskANN's decomposition maps almost one-to-one onto code kern already had. A
Vamana graph is a single-layer, alpha-pruned proximity graph; kern's HNSW
already implements the beam search, neighbour pruning, and heap machinery —
the deltas are dropping the layer hierarchy and putting adjacency on disk. The
shipped standalone index (`src/base/diskann.rs`) is exactly that:

- **Deterministic batch build** (`src/base/diskann.rs:151`): random regular
  graph, then two RobustPrune passes at alpha 1.0 and 1.2, fixed RNG seed, so
  the same corpus always yields the same index. Defaults: degree 32, build
  list 64 (`src/base/diskann.rs:26`).
- **Flat mmapped files** (`src/base/diskann.rs:302`): fixed-stride `f32`
  vectors in `vectors.bin`, fixed out-degree adjacency in `graph.bin`, both
  memory-mapped and size/bounds-validated on open so a truncated or corrupt
  index is rejected rather than read out of bounds. The OS page cache, not the
  heap, holds the vectors.
- **A pinned recall floor**: the build is held to recall@10 ≥ 0.90 against
  brute force by an inline test (`src/base/diskann.rs:454`) — the trade is
  named, not assumed.

The live wiring hides the choice behind a two-variant enum
(`src/base/vector_backend.rs:8`): `Resident(HnswIndex)` or
`Disk { snapshot, delta, tombstones }`. Callers cannot tell which they have.
`rebuild_index` makes the spill decision (`src/base/graph.rs:234`): once the
resident searchable-entity count exceeds `disk_threshold`, it builds a
snapshot under `<data_dir>/diskann/entity` and switches the entity index to
the disk variant. Any build or open failure logs and falls back to the
resident index (`src/base/graph.rs:294`) — a disk error never breaks the
graph.

Continuous ingest is handled with FreshDiskANN-style semantics rather than
rebuild-on-write. Every post-snapshot insert goes into a small in-RAM HNSW
delta and simultaneously tombstones its id against the snapshot
(`src/base/vector_backend.rs:68`), so a search over
`(snapshot − tombstones) ∪ delta` never serves an id twice
(`src/base/vector_backend.rs:92`). A tick task folds the delta back into a
fresh snapshot once it exceeds 10,000 entries, at most hourly
(`src/tick/pulse.rs:84`, constants at `src/base/constants.rs:107`), keeping
the delta bounded.

Worked example: a kern configured with `disk_threshold = 100_000` crosses
100,001 searchable entities. The next `rebuild_index` writes a Vamana snapshot
and the heap drops by the full vector set — queries now walk mmapped adjacency
and pay page faults instead of RAM. The day's new captures accumulate in the
delta and are searchable immediately; overnight the consolidation tick folds
them in. A kern that never crosses the threshold never touches any of this.

## What shipped versus what the note assumed

The research note has been updated in place and is close to reality; the
honest gaps:

- **Entity index only.** The GNN-vector and reason-edge indexes are always
  rebuilt resident (`src/base/graph.rs:227`), so a spilled kern still carries
  two in-RAM indexes that grow with the corpus. The ceiling is pushed back,
  not removed.
- **No product quantization.** The note's PQ-codes-in-RAM stage
  (`QuantizationMode::Pq`) was never built; modes remain `None`/`Int8`, with
  `Binary` implemented but rejected at parse (`src/quant.rs:7`). The snapshot
  mmaps full `f32` vectors — which already achieves the decision's goal of
  removing them from the resident heap; PQ is a later
  page-cache-footprint optimization, not a prerequisite, and its open
  questions (codebook training, drift on model swap) are still open.
- **Off by default is a posture, not an accident.** The default threshold is
  `KERN_CAP_DISABLED` (`src/base/graph.rs:169`) — never spill. Nothing
  auto-tunes it; an operator opts in per deployment via `[graph]
  disk_threshold`.
- **The note's original Phase 1 is obsolete**, as described above — the cold
  tier it targeted was replaced by the bounded LMDB scan before DiskANN
  landed there.

<Callout type="info" title="Where this sits in the pipeline">
The spill changes where dense-seed vectors live, and nothing else: the same
`VectorBackend` calls serve [retrieval](../concepts/retrieval.mdx)'s dense
seeding whether resident or spilled, filters included. Recall behaviour at the
boundary is governed by the 0.90 recall floor plus the exact in-RAM delta for
recent writes.
</Callout>
